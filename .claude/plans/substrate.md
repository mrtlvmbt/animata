# Substrate-слой animata — план перед интеграцией симуляции жизни

## Цель и границы

Построить **минимальный environment-substrate**, который будущая sim-of-life будет ЧИТАТЬ
(и для растительности — изменять), не строя при этом симуляцию погоды. Климат **статичный
per-column** (зафиксировано с пользователем). Погода / сезоны / ветер / суточный визуал —
**вне scope**, отдельная фаза после жизни.

Три substrate-фазы, каждая = `--no-ff` ветка к `main`, gated (clippy `-D warnings` + tests),
как и worldgen A–G. Порядок: S1 → S2 → S3 (зависимости снизу вверх).

### Что НЕ делаем (явно отложено)
- Динамическая погода (дождь/облака/фронты), сезоны, ветер.
- Суточный цикл день/ночь как ВИЗУАЛ (освещение/небо). Clock-значение в S2 — да, рендер-реакция — нет.
- Сама симуляция существ (метаболизм/поведение/spawning) — это следующая программа после substrate.
- Перестройка трав/деревьев-визуала под biomass (декорация-деревья остаются как есть в S3).

- Per-column query API `height_at/biome_at/is_water` есть, под `#[allow(dead_code)]` (terrain.rs:460).

- Главный цикл (main.rs:786): `let dt = get_frame_time();` уже есть; мирового времени/sim-tick нет.
- Hydrology (`compute`) — пример глобального однопроходного пасса при генерации; distance-to-water

## S1 — Environment-поля (queryable, резидентные per-column)

**Задача:** перестать выбрасывать климат; дать sim стабильный read-API под позицией существа.

**Изменения:**
1. `classify` вычисляет `temp` и `moist` **для ВСЕХ колонок** (вынести оба вычисления ВЫШЕ
   altitude-веток), возвращает их рядом с `(surf, biome)`. Логика выбора биома НЕ меняется —
   beach/mountain/snow по-прежнему игнорируют climate; temp/moist просто ещё и сохраняются.
   ⇒ нулевой риск регрессии классификации (та же ветвь, тот же результат биома).
2. `VoxelTerrain` получает резидентные `temp:Vec<u8>`, `moist:Vec<u8>` (квант `[0,1]→[0,255]`,
   `(v*255).round() as u8`). +2 B/кол ≈ +6.7 MB при ×16.
3. `water_dist:Vec<u8>` — расстояние до ближайшей water-колонки в колонках, saturating при 255.
   Однопроходный multi-source BFS в `generate` (как hydrology), вызов после того как `water`
   заполнен. +1 B/кол ≈ +3.4 MB.
4. `slope` — **НЕ хранить**, считать на запрос из `surf` соседей (max abs разность уровней с
   4-соседями, нормировка). Out-of-world сосед = собственная высота (нулевой вклад).
5. Публичный read-API (снять `#[allow(dead_code)]`, т.к. появляется реальный потребитель —
   но потребителя ещё нет в S1, см. ниже про мёртвый код):
   `temperature_at(x,y)->f32`, `moisture_at(x,y)->f32` (де-квант `/255.0`), `slope_at(x,y)->f32`,
   `water_dist_at(x,y)->u8`. Bounds: out-of-world → разумные дефолты (temp/moist 0.0, dist 255).

**Мёртвый код в S1:** в S1 ещё нет sim-потребителя этих геттеров. Чтобы не вводить
`#[allow(dead_code)]`-долг и не оставлять непроверенный API, S1 ОБЯЗАН включать потребление:
(а) dev-bridge `animata/status` отдаёт temp/moist/water_dist/slope под курсором/центром карты;
(б) `G`-debug-overlay уже есть (топо) — добавить режимы colourmap для temp и moist (циклом по `G`),
чтобы поля визуально верифицировались (метод «инструментируй поле» из worldgen). Геттеры тем
самым имеют живых вызывающих → без `dead_code`-аннотаций.

**Память (бюджет ×16):** модель растёт 7→10 B/кол (+temp+moist+water_dist = 3 B), +~11 MB → ~37 MB
(3.69M кол). Деревья-меши и так доминируют (стримятся). OK.

**Тесты:** temp/moist в `[0,1]`; полнота (нет «дыр» — каждая колонка имеет значение, включая
beach/mountain/snow, которые раньше не считали climate); квант-roundtrip в пределах 1/255;
water_dist=0 на water-колонках и монотонный рост от берега; slope=0 на плоскости, >0 на склоне;
детерминизм по seed. Headless (как существующие — без GL).

**Открытый вопрос для человека:** квантизация temp/moist в `u8` (1/255 ≈ 0.4%) против хранения
`f32` (×4 память). Рекомендация: `u8` — климат статичен и грубее этого порога, +10 MB vs +27 MB.

## S2 — Время / sim-clock (база `dt`-детерминизма)

**Задача:** дать симуляции монотонное мировое время и фиксированный шаг. Детерминизм/репро —
гарантия ТОЛЬКО на headless-пути фиксированного шага (`advance`); интерактивный путь — best-effort
(см. trade-offs, F2). Без рендер-реакции (день/ночь-визуал отложен).

**Изменения:**
1. `struct WorldClock { tick: u64, time_scale: f32, accum: f32, tick_len: f32 }`.
   - **`tick: u64`** — счётчик выполненных под-шагов = КАНОНИЧЕСКОЕ время. `sim_time` производное:
     `sim_time() -> f64 = tick as f64 * tick_len as f64`. (Целочисленный счётчик — нет f32/f64-дрейфа
     метки; ленивый реген S3 читает `tick` напрямую, см. F1.)
   - `time_scale` — sim-секунд на реальную секунду (по умолчанию 1.0; настраивается).
   - **Канонический детерм-API (реальный кодопуть, не свойство теста):**
     `advance(&mut self, n: u64)` — выполняет ровно `n` под-шагов (инкремент `tick`, вызов sim-tick).
     Headless-тесты и будущий sim зовут ЕГО. Отвязан от `get_frame_time()`.
   - **Интерактивная обёртка** над `advance`: `accum += dt * time_scale;
     let n = (accum / tick_len).floor().min(MAX_SUBSTEPS); accum -= n * tick_len; self.advance(n);`.
     Spiral-of-death guard: `MAX_SUBSTEPS` (напр. 8) clamp + сброс лишнего `accum` за порогом —
     это и есть источник best-effort-недетерминизма интерактива (F2).
2. `day_frac()->f32` = `(sim_time() / DAY_LEN).fract()` для будущей привязки (визуала пока нет).
   `DAY_LEN` — конст в `config.rs`.
3. HUD-readout (рядом с fps, toggle `I`): `sim_time`, `day_frac`, `tick#`.
4. Dev-bridge `animata/status`: `sim_time`, `day_frac`, `ticks`. Новый RPC `animata/set_timescale`.
5. Пауза: клавиша (напр. `P`) ставит `time_scale`-gate (часы стоят) — пригодится для отладки sim.

**Мёртвый код в S2:** sim-tick тела ещё нет. Чтобы счётчик-петля не была no-op-каркасом без
потребителя — она ИНКРЕМЕНТИРУЕТ `tick`-счётчик, который идёт в HUD и status. Тем самым петля
наблюдаема и тестируема (число тиков за N секунд при заданном time_scale предсказуемо).

**Тесты:** `advance(n)` даёт ровно `n` тиков и `sim_time()==n*tick_len` (канонический путь);
интерактивная обёртка при фиксированном `dt`/`time_scale` детерминирована; spiral-guard: гигантский
`dt` (lag-спайк) НЕ порождает > `MAX_SUBSTEPS` под-шагов; `day_frac` в `[0,1)`; пауза замораживает
`tick`. Headless (логика часов чистая, без macroquad).

**Открытый вопрос для человека:** `tick_len` (sim-разрешение). Рекомендация: `0.1 s` (10 тиков/sim-с)
как старт — компромисс точность/стоимость; настраиваемо.

## S3 — Растительность как ПОТРЕБЛЯЕМЫЙ ресурс (низ пищевой цепи)

**Задача:** резидентное per-column поле биомассы, которое sim читает, ДЕКРЕМЕНТИРУЕТ (выпас) и
которое РЕГЕНЕРИРУЕТ к biome-зависимой ёмкости. Первое динамическое поле.

**Изменения:**
1. `biomass:Vec<u8>` в `VoxelTerrain` (квант `[0,cap]→u8`). Ёмкость `carrying_capacity(biome,
   moist, temp)` — таблично по biome (вода/пустыня/скала низкие; forest/jungle/plains высокие),
   модулировано moisture/temp из S1. **Начальная биомасса = `cap`** (зрелая растительность на
   момент worldgen); `last_update=0`. С законом ниже из `b0=cap` нетронутая колонка стабильна
   (`elapsed`-рост не превышает cap). +1 B/кол ≈ +3.7 MB.
2. **Ленивая (амортизированная) регенерация — НЕ глобальный тик.** При ×16 пройтись по 3.69M
   колонок каждый sim-tick недопустимо. Вместо этого per-column **`last_update:Vec<u32>`** —
   значение `tick`-счётчика (S2) на момент последнего касания. Текущая биомасса вычисляется ЛЕНИВО
   при запросе/потреблении: `elapsed = (tick - last_update[i] as u64) as f64 * tick_len`;
   `regrow(b, cap, elapsed)` **законом линейный-с-насыщением** `b' = cap − (cap − b)·e^{−r·elapsed}`
   (замкнутая форма ⇒ ленива; монотонна; не превышает cap; **восстанавливается из 0**:
   `b=0 ⇒ cap·(1−e^{−r·elapsed})` — нет неподвижной точки, нет вечных плешей под пасущимся стадом);
   затем `last_update[i] = tick as u32`. **Целые тики ⇒ нет f32-дрейфа метки** (F1: f64-точность
   часов не выкидывается в записи). Колонки, которых sim не касается, не стоят ничего.
   +4 B/кол ≈ +14.8 MB (`u32`, 3.69M кол). `u32`-тики при `tick_len=0.1` переполняются через
   ~13.6 лет sim-времени — не проблема; `u16` (мёртв, wrap ~1.8 ч) отвергнут.
3. API: **`tick` передаётся ЯВНЫМ аргументом**, НЕ кешируется в `VoxelTerrain` (F4: иначе время
   втекает неявно — кеш, обновляемый системой после `graze`, даёт тихий рассинхрон `elapsed`).
   `biomass_at(x,y,tick)->f32` (ленивый пересчёт, БЕЗ записи — read-only оценка),
   `graze(x,y,amount,tick)->f32` (ленивый пересчёт + запись `last_update=tick` + декремент,
   возвращает реально снятое). Растительность НЕ растёт под водой (`is_water` ⇒ cap 0).
4. **Визуал-деревья остаются декорацией** (S3 не перестраивает меши под biomass). Связь
   «дерево ⇔ biomass» — будущая фаза. S3 даёт только sim-поле + закон.

**Потребитель в S3 (против мёртвого кода):** настоящего травоядного ещё нет. Чтобы `graze`/закон
регена были живыми и проверяемыми: dev-bridge RPC `animata/graze {x,y,amount}` (снимает биомассу)
+ `animata/biomass {x,y}` (читает); `G`-overlay получает colourmap биомассы. Это даёт ручную
числовую верификацию закона до появления sim (метод worldgen: инструментируй поле).

**Память (итог ×16):** поверх базы 7 B/кол S1 добавляет 3 B (temp+moist+water_dist), S3 — 5 B
(biomass 1 + last_update 4) → модель **15 B/кол × 3.69M ≈ ~55 MB** (+~29 MB к базовым ~26).
В абсолюте ок (меши доминируют и стримятся).

**Тесты:** свежая колонка → биомасса = `cap`; `graze` сверх наличного снимает ≤ наличное;
**`graze` ДО 0, затем `elapsed` → биомасса восстанавливается из 0 к `cap`** (явный анти-плешь тест,
ловит неподвижную точку — был бы провален логистикой); рост монотонен, не превышает `cap`;
water-колонка всегда biomass 0; ленивый путь эквивалентен пошаговому (один `elapsed=10·tick` ==
10×`elapsed=tick` в пределах кванта); **long-chain (F5): ≥100 циклов graze+реген одной клетки,
assert |итог − аналитика| ≤ k/255** (квант-шум ограничен, не дрейфует); детерминизм. Headless.

**Решённые вопросы (не требуют человека):**
- Закон регена = **линейный-с-насыщением** (F3-фикс): восстанавливается из 0, без неподвижной
  точки/плешей, дешевле логистики. Если ПОЗЖЕ захочется именно S-кривой — логистика с seed-floor
  `b0_eff=max(b0,ε·cap)` (лишний параметр), но плеши того не стоят на старте.
- Хранение метки времени = `u32`-тики (F1).

## Последовательность и gating

1. **S1** ветка → clippy+tests зелёные → визуальная проверка temp/moist overlay локально → PR в `main`.
2. **S2** ветка (поверх S1) → tests → PR.
3. **S3** ветка (поверх S2; ленивый реген использует `sim_time` из S2) → tests → визуальная проверка
   biomass overlay + ручной `graze` через bridge → PR.

После S1–S3 substrate готов: следующая программа — сама симуляция существ (читает temp/moist/slope/
water_dist/biomass, пишет `graze`, тикает на `WorldClock`).

## Принятые операционные риски (trade-offs)

- **F2 [severity: tradeoff] — интерактивный детерминизм best-effort.** Spiral-guard
  (`MAX_SUBSTEPS` clamp + сброс лишнего `accum`) делает число тиков за кадр зависимым от реального
  lag-спайка, поэтому два интерактивных прогона одного seed на разном фрейм-пейсинге могут разойтись.
  Сознательный компромисс real-time vs репродьюс — без записи входов / фикс-`dt` его не снять.
  Гарантия детерминизма/репро дана ТОЛЬКО на headless-пути `advance(n)` (где живут тесты и будущий
  sim-batch); интерактив честно помечен best-effort. Будущий sim-автор не должен опираться на
  воспроизводимость интерактивного прогона.
- **water_dist u8 saturating@255 [severity: tradeoff, критик «принят»].** Внутренние плато-регионы
  далеко от воды читаются плоским 255 на 1920-широкой карте. Для sim («далеко = далеко») приемлемо;
  градиент у берега (где он нужен для гидрации) сохранён точным.
- **F5 [severity: style] — квант-пол биомассы `cap/255`.** Мелкозернистый выпас ниже `cap/255`
  теряется в округлении `u8`. НЕ free-food (graze возвращает реально снятое; sim видит 0 при
  недокусе), шум ограничен `1/255`/запись и зажат в `[0,cap]` — не дрейфует. Long-chain тест выше
  это сторожит. Если sim-автору понадобится тонкий метаболизм — поднять биомассу до `u16` (квант
  настраиваем, как и temp/moist). Принят: на старте `u8` достаточно.
- **Плотный `last_update:Vec<u32>` vs `HashMap` тронутых [severity: tradeoff].** Плотный `Vec`
  (+14.8 MB) на разреженно-касаемом поле «дороже» `HashMap` только-тронутых (~−15 MB при локальных
  стадах). Принят плотный `Vec`: проще, без edge-case на ёмкость карты, память в бюджете (~55 MB).
  Перейти на `HashMap` если профиль покажет реальную разреженность.
